## A. 業務邏輯規則（優先級最高 — C）

### A1. 台灣發票稅務規則

#### 進項稅計算
- **公式**：`進項稅金額 = 發票金額 × 5%`（5% 稅率）
- **精度**：DECIMAL(19,2)，向下取整（銀行家舍入）
- **應用場景**：
  - 三聯式發票：進項稅 = 金額 × 0.05
  - 二聯式發票：金額已含稅，需先拆分金額和稅額

#### 三聯式 vs 二聯式發票
- **三聯式**（不含稅）：
  - 儲存的金額 = 發票金額（不含稅）
  - 進項稅 = 金額 × 0.05
  - 用途：B2B（對方公司有統編）

- **二聯式**（含稅）：
  - 儲存的金額 = 含稅金額
  - 拆分邏輯：稅前金額 = 含稅金額 / 1.05；稅額 = 含稅金額 - 稅前金額
  - 用途：B2C（零售、個人）

#### 進項稅折讓規則
- **定義**：購買發票的金額調整（退貨、折讓）
- **申報期限**：當期內必須申報（無延後例外，根據 2026 稅務局規定）
- **記錄位置**：`TW_InvoiceAdjustment` 表
- **超期警告**：超過 2 個申報期限未申報時，系統應警告

#### 進項稅可扣抵期限
- **期限**：開立日期後 10 年內
- **警告**：距離期滿 90 天內，系統應警告（IsExpiryWarning）
- **不可扣抵費用**：
  - 交際應酬費
  - 部分國內出差費
  - 自用小客車相關費用（某些項）

#### 發票號碼規則
- **連續性**：必須連續，允許跳號但需報告
- **格式**：字軌（2 個大寫字母） + 號碼（7-8 位數字）
- **範例**：`AA0000001` ~ `AA9999999`
- **每個字軌**：獨立的號碼範圍
- **號碼檢查**：系統應檢查連續性，遺失號碼記錄到 TW_InvoiceAdjustment

#### 兼營營業人稅額調整
- **定義**：同時進行應稅和免稅營業的企業
- **調整時機**：年度結算（12 月）
- **調整公式**：
  ```
  可扣抵進項稅 = 總進項稅 × (應稅銷售額 / 總銷售額)
  ```
- **記錄**：`TW_TaxStatement.MixedBusinessRatio`

### A2. 發票開立日期 → 申報期間對應

台灣營業稅申報為雙月制，根據開立日期決定申報期間：

| 開立月份 | 申報期間 | 申報截止 |
|---------|--------|--------|
| 1-2 月  | 第 1 期  | 3/15   |
| 3-4 月  | 第 2 期  | 5/15   |
| 5-6 月  | 第 3 期  | 7/15   |
| 7-8 月  | 第 4 期  | 9/15   |
| 9-10 月 | 第 5 期  | 11/15  |
| 11-12 月| 第 6 期  | 次年 1/15 |

**實現**：根據 `DateInvoiced` 月份自動決定 `TaxReportingPeriod`

## B. 設計稿對齐規則（優先級中 — A）

### B1. 發票建立流程（API）

**流程**：選字軌 → 分配號碼 → 輸入日期 → 計算稅額 → 生成 QR → 建立記錄

**API 端點**（預期）：

```
POST /invoices
{
  "documentType": "SALES_TRIPART" | "SALES_BIPART",
  "dateInvoiced": "2026-03-25",
  "amount": 1000.00,
  "customerId": "C_Customer_ID"
}

Response (201 Created):
{
  "id": "uuid",
  "invoiceNumber": "AA0000001",
  "prefix": "AA",
  "number": 1,
  "dateInvoiced": "2026-03-25",
  "amount": 1000.00,
  "taxAmount": 50.00,
  "taxReportingPeriod": "1",  // 1-6
  "qrCode": "...",
  "status": "DRAFT",
  "createdAt": "2026-03-25T10:00:00Z"
}
```

**檢查點**：
- [ ] 返回欄位是否完整？
- [ ] 稅額計算是否正確？
- [ ] QR Code 是否生成？

### B2. 發票列表 API

**預期返回格式**：

```json
{
  "data": [
    {
      "id": "...",
      "invoiceNumber": "AA0000001",
      "amount": 1000.00,
      "taxAmount": 50.00,
      "dateInvoiced": "2026-03-25",
      "status": "DRAFT" | "POSTED" | "CANCELLED",
      "taxReportingPeriod": 1
    }
  ],
  "pagination": { "total": 100, "page": 1, "size": 20 }
}
```

## D. 代碼結構與命名規範

### D1. Package 結構

```
com.example.tw.invoice.tax/
├── domain/
│   ├── entity/          # JPA entities (MInvoicePrefix, TW_TaxStatement)
│   ├── value/           # 值對象 (TaxRate, InvoiceNumber)
│   └── model/           # 業務模型 (Invoice, TaxAdjustment)
├── service/             # 業務邏輯
├── repository/          # 數據訪問
├── validator/           # 驗證邏輯（純 Java）
├── event/               # 事件定義
├── exception/           # 自定義異常
├── dto/                 # 數據傳輸對象
├── config/              # 配置類
└── resource/            # REST 資源 (JAX-RS)
```

### D2. 命名規範

- 類名：PascalCase (InvoicePrefix, TaxStatement)
- 方法名：camelCase，動詞開頭 (getTaxAmount, calculateTax)
- 變數名：camelCase (invoiceNumber, taxAmount)
- 常數：UPPER_SNAKE_CASE (MAX_INVOICE_NUMBER, DEFAULT_TAX_RATE)
- DB 表名：snake_case (tw_invoice_prefix, c_invoice)

## E. 測試規範

### E1. 單元測試

**最低標準**：覆蓋度 80%+

**必須測試的場景**：

- 稅額計算：
  - [ ] 正常金額
  - [ ] 零金額（應拒絕？）
  - [ ] 最大值
  - [ ] 負數（應拒絕）

- 發票號碼分配：
  - [ ] 連續分配
  - [ ] 號碼跳號檢測
  - [ ] 同字軌重複檢查

- 日期 → 申報期間：
  - [ ] 各月份對應期間
  - [ ] 邊界情況（月末日）

### E2. 集成測試

**必須有**（不只是 Mock）：

- 真實 Database 的發票儲存和取回
- 稅額計算的準確性（涉及舍入）
- 發票號碼連續性檢查

## G. 術語定義

| 術語 | 定義 |
|------|------|
| 字軌 | 發票的 2 位字母前綴（如 AA、AB） |
| 號碼 | 發票的 7-8 位數字尾部（如 0000001） |
| 進項稅 | 購入商品/服務的稅款（可申報扣抵） |
| 銷項稅 | 銷售商品/服務的稅款（需申報繳納） |
| 應稅銷售額 | 需徵收 5% 營業稅的銷售收入 |
| 免稅銷售額 | 不徵收營業稅的銷售收入（如出口） |
| 兼營營業人 | 同時進行應稅和免稅營業的企業 |
| 進項稅折讓 | 購買發票的金額調整（退貨、折讓） |
